# AORR 상태 머신 설계서

대상: GitHub Pages에서 실행 가능한 개인 프로페셔널 정적 웹사이트

제약:
- 백엔드 서버 없음
- 루트 디렉토리에 `index.html`, `styles.css`, `script.js`가 존재해야 함
- 게임 코드는 `script.js` 내부 또는 별도 JavaScript 파일로 구현
- 모바일과 데스크톱 모두 지원
- 코드 수정, 테스트 실행, 배포는 이 문서 범위 밖이며 실제 수행 대상이 아님

## 1. Target

### 프로페셔널 웹사이트 개발 목표
- 개인 소개, 경력 요약, 프로젝트, 연락처를 담은 프로페셔널 웹사이트를 만든다.
- 첫 화면에서 방문자가 누구의 사이트인지 즉시 이해할 수 있어야 한다.
- 정적 웹사이트로 유지하며, 서버 의존성을 두지 않는다.

### GitHub Pages 배포 목표
- GitHub Pages에서 정상적으로 열리는 정적 사이트를 목표로 한다.
- 상대 경로와 정적 자산만 사용한다.
- 빌드 도구가 필요하다면 선택 사항으로 두되, 최종 산출물은 정적 HTML/CSS/JS로 동작해야 한다.

### 입력 자료
- 저장소의 기존 파일 구조
- 개인 이름, 소개 문구, 경력, 프로젝트, 연락처 [사람 확인 필요]
- 브랜드 톤, 색상 선호, 사진/로고 사용 여부 [사람 확인 필요]
- Games 탭에 넣을 지렁이 게임 요구사항
- GitHub Pages 대상 저장소 설정 정보 [사람 확인 필요]

### 필수 페이지와 섹션
- 상단 내비게이션
- Hero 또는 소개 섹션
- About 또는 Profile 섹션
- Projects 또는 Works 섹션
- Contact 섹션
- Games 섹션 또는 Games 페이지
- 푸터

### Games 탭 및 지렁이 게임 요구사항
- 상단에 `Games` 탭이 있어야 한다.
- 키보드 입력으로 조작 가능한 지렁이 게임이 있어야 한다.
- 모바일 터치로도 조작 가능해야 한다.
- 점수, 게임오버, 재시작이 있어야 한다.
- 게임은 HTML/CSS/JavaScript만으로 동작해야 한다.
- Step 1에 `[게임 추가 기능:]`이 포함되어 있다면, 해당 추가 기능은 아래 게임 루프에 반영한다. 다만 현재 정보로는 구체 항목이 없으므로 [사람 확인 필요]로 둔다.

### 데스크톱 및 모바일 완료 기준
- 데스크톱에서 레이아웃이 깨지지 않는다.
- 모바일에서 가로 스크롤이 발생하지 않는다.
- 메뉴와 게임 조작 요소를 손가락으로 충분히 누를 수 있다.
- 콘솔 오류가 없다.
- GitHub Pages에서 정적 자산 경로가 깨지지 않는다.

## 2. Act

### 한 번의 개발 루프에서 수행할 최소 작업
- 실패 원인 하나만 선택한다.
- 관련된 최소 파일만 수정한다.
- 수정 후 같은 검증 절차를 다시 실행한다.
- 통과한 기능에 대한 회귀 확인을 병행한다.

### 수정 가능한 파일 범위
- 루트의 `index.html`
- 루트의 `styles.css`
- 루트의 `script.js`
- 게임 분리를 선택한 경우 루트의 추가 JavaScript 파일
- 정적 자산 파일만 추가 가능
- GitHub Pages용 설정 파일은 필요할 때만 추가

### 생성할 수 있는 파일
- `index.html`
- `styles.css`
- `script.js`
- `game.js` 또는 `snake.js` 같은 보조 JavaScript 파일
- 이미지, 아이콘, 폰트는 정적 자산으로만 추가

### 실행 가능한 로컬 검증 명령어
- 파일 존재 확인
- HTML 구조 검토
- CSS 구문 검토
- JavaScript 구문 검토
- 로컬 정적 서버 실행
- 브라우저 수동 확인
- 모바일 뷰포트 점검

검증 명령은 환경에 따라 달라질 수 있으므로 구체적인 도구 이름은 저장소 확인 후 확정한다.

## 3. Observe

관찰 항목:
- 파일 생성 여부
- HTML 오류
- CSS 반응형 오류
- JavaScript 오류
- 로컬 웹서버 응답
- 브라우저 콘솔 오류
- 데스크톱 화면
- 모바일 화면
- 키보드 게임 조작
- 터치 게임 조작
- GitHub Pages 호환성

관찰 규칙:
- 먼저 실패를 재현한다.
- 다음에 변경 전후 차이를 최소 단위로 확인한다.
- 브라우저에서 확인한 현상은 콘솔 메시지와 함께 기록한다.

## 4. Reason

실패 원인 분류 기준:

| 분류 | 의미 |
|---|---|
| HTML_STRUCTURE | 문서 구조, 태그 중첩, 시맨틱 구조 문제 |
| CSS_RESPONSIVE | 레이아웃, 브레이크포인트, 모바일 대응 문제 |
| JAVASCRIPT | 일반 JavaScript 문법, 상태, 비동기, 이벤트 문제 |
| GAME_LOGIC | 이동, 충돌, 점수, 게임오버 등 게임 규칙 문제 |
| GAME_CONTROL | 키보드, 터치, 제스처, 입력 반응 문제 |
| CONTENT | 이름, 소개, 경력, 프로젝트 등 개인 콘텐츠 문제 |
| TEST | 검증 명령, 테스트 절차, 관찰 방법 문제 |
| ENVIRONMENT | 로컬 실행 환경, 브라우저, 경로, 도구 문제 |
| GITHUB_PERMISSION | 저장소 권한, 토큰, 브랜치 접근, Pages 설정 권한 문제 |
| DEPLOYMENT | GitHub Pages 배포, 정적 호스팅, 경로 문제 |
| UNKNOWN | 원인을 특정할 수 없음 |

분류 규칙:
- 하나의 실패는 하나의 주 분류로만 태깅한다.
- 원인이 복합적이면 가장 먼저 해결해야 할 원인을 주 분류로 선택한다.
- 확신이 없으면 `UNKNOWN`으로 분류하고 추가 관찰을 진행한다.

## 5. Repeat

반복 규칙:
- 한 번에 하나의 실패 원인만 수정한다.
- 관련된 최소 파일만 변경한다.
- 수정 후 동일한 Verifier를 다시 실행한다.
- 기존에 통과한 기능에 대한 회귀 테스트를 함께 실행한다.
- 같은 오류 fingerprint가 두 번 반복되면 반복을 중단하고 `BLOCKED` 또는 `HITL_REQUIRED`로 전환한다.

## 6. Stop

중단 조건:
- 전체 테스트가 통과한 경우
- 최대 Retry에 도달한 경우
- 동일한 오류 fingerprint가 2회 반복된 경우
- 개인정보나 콘텐츠 확인이 필요한 경우
- GitHub 인증 또는 배포 권한 문제가 발생한 경우

권장 최대 Retry:
- 일반 UI 또는 콘텐츠 실패: 2회
- 게임 로직 실패: 2회
- 권한/환경 실패: 1회 재시도 후 중단 가능

## 7. Human-in-the-loop

다음 상황에서는 사람 확인이 필요하다.
- 이름, 소개, 경력, 프로젝트 등 개인 콘텐츠가 불명확한 경우
- 기존 콘텐츠 삭제가 필요한 경우
- 외부 분석 도구나 외부 서비스를 추가해야 하는 경우
- GitHub 저장소 설정을 변경해야 하는 경우
- 요구사항이 충돌하는 경우
- `Games` 탭에 들어갈 추가 기능이 구체화되지 않은 경우

표기 규칙:
- 문서나 구현 과정에서 불명확한 항목은 `[사람 확인 필요]`로 적는다.

## 8. AORR 상태 정의

| 상태 | 의미 |
|---|---|
| READY | 다음 루프를 시작할 준비가 됨 |
| ACTING | 파일 수정 또는 생성 작업 수행 중 |
| VERIFYING | 검증, 관찰, 수동 확인 수행 중 |
| RETRYING | 실패 원인을 좁혀 재시도 중 |
| PASSED | 현재 루프가 통과됨 |
| DEPLOY_READY | 배포 가능한 상태이며 배포 전 마지막 확인이 끝남 |
| DEPLOYING | GitHub Pages 배포를 실행 중 |
| DEPLOYED | GitHub Pages 배포가 완료됨 |
| BLOCKED | 기술적 또는 운영상 진전이 막힘 |
| HITL_REQUIRED | 사람 확인이 필요한 상태 |

## 10. 상태 전이 규칙

- `READY`는 다음 루프의 시작 상태다.
- `ACTING`은 변경을 가하는 상태다.
- `VERIFYING`은 관찰과 검증 상태다.
- `PASSED`는 해당 루프의 성공 종료다.
- `RETRYING`은 같은 실패 원인을 좁혀 다시 시도하는 상태다.
- `DEPLOY_READY`는 구현이 끝나고 배포만 남은 상태다.
- `DEPLOYING`은 실제 Pages 배포 상태다.
- `DEPLOYED`는 최종 완료 상태다.
- `BLOCKED`는 더 이상 안전하게 전진할 수 없는 상태다.
- `HITL_REQUIRED`는 사람의 결정을 먼저 받아야 하는 상태다.

## 11. 권장 실행 순서

3. 프로페셔널 콘텐츠 영역
4. 반응형 내비게이션
5. Games 탭
6. 지렁이 게임 핵심 로직
7. 키보드 조작
8. 모바일 터치 조작
9. 게임 UI 및 점수
10. 접근성과 반응형 검증

12. 배포

## 12. 첫 루프로 적합한 상태

가장 안전한 첫 루프는 `저장소 및 기존 파일 확인`이다.

이유:
- 현재 저장소 구조가 아직 완전히 확정되지 않았다.
- 기존 변경이 있을 수 있어 충돌을 피해야 한다.
- 게임과 반응형 설계는 저장소 스택에 따라 구현 방식이 달라진다.

## 13. 출력 규칙

- 설계 문서는 구현보다 먼저 만든다.
- 확정되지 않은 항목은 추측하지 않고 `[사람 확인 필요]`로 남긴다.
- 구현 단계에서는 이 문서의 상태 전이와 중단 조건을 그대로 따른다.

## 14. Self-Correcting TDD Loop

이 섹션은 실제 구현 단계에서 사용할 Verifier 중심의 Self-Correcting TDD 루프를 정의한다.
현재 이 저장소는 정적 웹사이트 소스가 아직 없으므로, 첫 실패 상태는 `루트 필수 파일 부재`로 간주한다.

### 14.1 현재 환경에서 확인된 검증 도구

| 도구 | 확인 결과 | TDD에서의 역할 | 비고 |
|---|---|---|---|
| `Get-ChildItem`, `Test-Path`, `Get-Content` | 사용 가능 | 파일 존재, 경로, 내용 확인 | PowerShell 기준 |
| `git` | 사용 가능 | 변경 추적, 회귀 비교, 경로 확인 | `git status`, `git diff --check` 같은 기본 검증에 사용 |
| `python` 3.14.150.1013 | 사용 가능 | `python -m http.server` 로컬 서버 | 정적 사이트 응답 검증에 사용 |
| `node` 24.11.0.0 | 사용 가능 | `node --check` 등 JavaScript 구문 검증 | 프로젝트 내 JS 파일이 생긴 뒤 사용 |
| `npm`, `npx` | 사용 가능 | 패키지 기반 검증 도구 실행 | 현재 저장소에 `package.json`은 없으므로 임의 명령을 만들지 않음 |
| `curl`, `Invoke-WebRequest` | 사용 가능 | HTTP 응답 확인, 외부 다운로드 | 외부 다운로드는 필요할 때만 |
| `claude.exe` 2.1.208 (Claude Code) | 사용 가능 | 독립 Verifier, 진단, 재현 프롬프트 실행 | 실제 모델 사용은 인증 상태와 세션 설정에 따름 |

### 14.2 현재 저장소 구조와 초기 실패 상태

현재 확인된 파일은 다음과 같다.

- `AORR.md`
- `README.md`
- `github_token.txt`
- `claude-code-cli/claude_auto_install.md`
- `claude-code-cli/env_settings.txt`

현재 확인되지 않은 파일:

- `index.html`
- `styles.css`
- `script.js`
- 게임용 보조 JavaScript 파일
- `package.json`

따라서 초기 TDD는 다음 실패를 먼저 해결하는 형태로 시작한다.

- 루트 `index.html` 부재
- `styles.css` 연결 부재
- `script.js` 연결 부재
- GitHub Pages용 상대 경로 기반 정적 진입점 부재

### 14.3 Claude Code CLI 모델 선택 정책

`claude --help`로 확인된 사실:

- `claude` CLI는 `--model <model>` 옵션을 지원한다.
- 도움말은 모델 alias와 full model name을 모두 받을 수 있음을 보여준다.
- 현재 CLI 버전은 `2.1.208 (Claude Code)`다.

다만 이 문서 작성 시점에는 인증된 비대화형 호출을 수행하지 않았으므로, 이 환경에서 **Sonnet 5가 실제로 사용 가능한지**는 아직 검증되지 않았다.

실행 시 정책:

1. 먼저 `claude doctor`로 설치 상태를 확인한다.
2. 이후 실제 Verifier 세션에서 `--model sonnet` 또는 계정이 허용하는 정확한 Sonnet 모델명을 시도한다.
3. 성공한 경우, 로그에 **실제 사용된 모델 문자열**을 그대로 기록한다.
4. Sonnet 5가 불가하면, CLI 또는 계정이 허용하는 가장 가까운 Sonnet 모델을 사용하고 동일하게 실제 문자열을 기록한다.
5. 모델 선택이 불명확하면 `HITL_REQUIRED`로 중단한다.

### 14.4 Verifier 우선순위

각 개발 루프는 다음 순서로 검증한다.

1. 파일 존재와 경로 확인
2. HTML 구조 확인
3. CSS 반응형 확인
4. JavaScript 구문 및 콘솔 확인
5. 게임 동작 확인
6. 로컬 서버 응답 확인
7. 브라우저 viewport 확인
8. GitHub Pages 호환성 확인

원칙:

- 한 Verifier는 한 번에 한 실패 원인만 판별한다.
- 같은 Verifier를 수정 후 반드시 다시 실행한다.
- 통과한 항목은 최소 회귀 세트로 재확인한다.

### 14.5 Self-Correcting 상태 머신

| 상태 | 목적 | 입력 | Act | Observe | 종료 조건 | 다음 상태 |
|---|---|---|---|---|---|---|
| READY | 다음 실패를 고를 준비 | 최신 실패 로그, 현재 파일 상태 | 검증 대상 1개 선택 | 우선순위와 범위 확정 | 수정 대상이 1개로 좁혀짐 | ACTING |
| ACTING | 최소 수정 | 실패 원인 1개, 관련 파일 1개 이상 | 필요한 파일만 수정 | 변경 범위가 최소인지 확인 | 수정이 완료됨 | VERIFYING |
| VERIFYING | 같은 Verifier 재실행 | 수정된 파일, 동일 검증 명령 | 같은 검증 명령 실행 | pass/fail, 콘솔, 응답, 화면 | 통과 또는 동일 실패 재발 | PASSED 또는 RETRYING |
| RETRYING | 동일 원인 재수정 | 실패 로그, fingerprint, 가설 | 같은 원인만 다시 수정 | 실패가 같은지 비교 | fingerprint가 바뀌거나 2회 반복됨 | PASSED, BLOCKED, HITL_REQUIRED |
| PASSED | 해당 루프 종료 | 성공한 검증 결과 | 회귀 세트 실행 준비 | 성공 기록 저장 | 다음 루프로 넘어갈 수 있음 | READY |
| BLOCKED | 환경/권한/중복 실패 | 반복 실패, 권한 문제 | 코드 수정 중단 | 문제 원인이 코드 밖인지 확인 | 더 이상 진전 불가 | STOP |
| HITL_REQUIRED | 사람 확인 필요 | 콘텐츠, 정책, 충돌 | 사용자 확인 대기 | 확인 질문 정리 | 응답이 올 때까지 중지 | STOP |

### 14.6 실패 로그 템플릿

실패가 발생하면 아래 필드를 반드시 기록한다.

| 필드 | 내용 |
|---|---|
| 실행 명령어 | 실제 실행한 명령 전체 |
| exit code | 종료 코드 |
| 실패한 검증 항목 | 어떤 Verifier가 실패했는지 |
| 핵심 오류 메시지 | 콘솔/브라우저/CLI의 핵심 메시지 |
| 관련 파일과 라인 | `file:line` 형식 |
| 브라우저 콘솔 메시지 | 콘솔에 나온 경고/오류 |
| 오류 fingerprint | 재발 판별용 짧은 해시 문자열 |
| 분류 | `HTML_STRUCTURE`, `CSS_RESPONSIVE`, `JAVASCRIPT`, `GAME_LOGIC`, `GAME_CONTROL`, `CONTENT`, `TEST`, `ENVIRONMENT`, `GITHUB_PERMISSION`, `DEPLOYMENT`, `UNKNOWN` |
| 가설 | 왜 실패했는지에 대한 1문장 가설 |
| 변경 파일 | 이번 Retry에서 수정한 최소 파일 |
| 재검증 결과 | 같은 Verifier 재실행 결과 |

권장 fingerprint 형식:

`<classification>|<file:line>|<error signature>|<viewport or command>`

예:

`JAVASCRIPT|script.js:42|Cannot read properties of null|desktop`

### 14.7 Retry 정책

- 하나의 오류에 대해 최대 3회까지 시도한다.
- 동일한 오류 fingerprint가 2회 반복되면 즉시 중지한다.
- 각 Retry는 한 가지 원인만 다룬다.
- 관련 없는 파일은 건드리지 않는다.
- 테스트 삭제, 검증 기준 완화, 외부 프레임워크 전환은 금지한다.
- 이미 통과한 기능은 회귀 세트로 재확인한다.
- 환경 또는 권한 문제는 코드 수정으로 우회하지 않는다.

| 항목 | Verifier | 실패 판정 |
|---|---|---|
| 루트 `index.html` | 파일 검증 | 루트가 아니면 실패 |
| 정적 상대 경로 | 문서 검증 | 절대 로컬 경로 사용 시 실패 |
| 서버 전용 기능 미사용 | 코드 검증 | 백엔드 전용 API 의존 시 실패 |
| 로컬 파일 시스템 의존성 미사용 | 코드 검증 | `file://` 전용 로직이 있으면 실패 |
| 백엔드 API 의존성 미사용 | 기능 검증 | 외부 API가 필수면 실패 |

### 14.9 반복 시 회귀 규칙

각 Retry 이후에는 다음 회귀 세트를 최소로 다시 확인한다.

1. `index.html` 존재
2. `styles.css` 연결
3. `script.js` 연결
4. 브라우저 콘솔 오류 없음
5. 로컬 서버에서 페이지가 열린다
6. 수정한 영역의 직전 통과 기능

게임 변경이 있었다면 추가 회귀:

1. 점수 증가
2. 게임오버
3. 키보드 조작
4. 모바일 조작
5. Games 탭 중복 실행 방지

### 14.10 권장 실행 순서

1. 저장소 및 기존 파일 확인
2. 기본 파일 검증
3. HTML 검증
4. CSS 검증
5. JavaScript 검증
6. 로컬 실행 검증
7. 브라우저 검증
8. 지렁이 게임 검증
9. GitHub Pages 호환성 검증
10. 배포 준비

### 14.11 TDD 루프의 종료 조건

- 모든 Verifier가 통과하면 종료한다.
- 같은 fingerprint가 두 번 반복되면 종료한다.
- 최대 Retry 3회에 도달하면 종료한다.
- 콘텐츠, 권한, 배포, 환경 문제는 `HITL_REQUIRED` 또는 `BLOCKED`로 전환 후 중지한다.

### 14.12 실행 시 기록해야 할 최소 산출물

- 사용한 실제 Verifier 명령
- exit code
- 실패 항목
- 핵심 오류 메시지
- 관련 파일과 라인
- 브라우저 콘솔 메시지
- 오류 fingerprint
- 수정한 파일 목록
- 재검증 결과
- 실제 사용한 Claude Code 모델 문자열
